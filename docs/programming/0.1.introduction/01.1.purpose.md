# Propósito del Documento

Este documento no es una lista de reglas porque sí. Es el acuerdo explícito de cómo trabajamos en SCISA: qué esperamos del código que entregamos, cómo nos comunicamos, y cómo garantizamos que el sistema que construimos hoy no sea un problema para el equipo de mañana.

---

## ¿Por qué existe este documento?

El software no lo escribe una sola persona. Lo escribe un equipo y un equipo sin acuerdos produce sistemas que se parecen a los acuerdos que tiene: ninguno.

Los lineamientos aquí descritos existen para resolver tres problemas concretos:

1. **Consistencia**: Que el código de cualquier desarrollador del equipo pueda ser leído, entendido y modificado por cualquier otro sin fricción.

2. **Calidad sostenida**: Que las decisiones técnicas se tomen con criterio y se documenten, en lugar de repetir los mismos errores o reinventar soluciones ya resueltas.

3. **Velocidad a largo plazo**: Un equipo que mantiene estándares altos desde el inicio entrega más rápido en el mediano y largo plazo. La deuda técnica acumulada es el freno más costoso que existe.

---

## ¿Qué NO es este documento?

- No es un checklist de burocracia que hay que cumplir para que te aprueben el PR.
- No es inmutable ni sagrado — si algo está mal, se cambia (ver [1.4 Cómo mantener actualizado este documento](01.4.maintaining.md)).
- No es un sustituto del criterio: si una situación no está cubierta aquí, usa juicio profesional y documenta tu decisión.
