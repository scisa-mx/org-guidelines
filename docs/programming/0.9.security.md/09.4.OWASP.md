# OWASP Top 10

El OWASP Top 10 es un estándar internacional que identifica las 10 vulnerabilidades más críticas en aplicaciones web.

Está publicado por la organización sin fines de lucro:

[OWASP](https://owasp.org/) (Open Web Application Security Project)

Su objetivo es ayudar a las organizaciones a:

- Reducir riesgos de seguridad
- Adoptar buenas prácticas en desarrollo seguro
- Priorizar esfuerzos de mitigación
- Crear cultura de seguridad en equipos técnicos

Apegarnos a estos estandares es fundamental para proteger nuestras aplicaciones y datos contra ataques comunes. No es solo para desarrolladores. Involucra: Arquitectura, DevOps, QA, Líderes técnicos, Seguridad, Gestión de riesgos.

## Broken Access Control
### Descripción
Ocurre cuando los usuarios pueden realizar acciones o acceder a recursos sin tener los permisos adecuados.
Es uno de los riesgos más explotados en aplicaciones modernas (APIs, microservicios, SPAs).

Incluye:

- Acceso a endpoints administrativos
- IDOR (Insecure Direct Object References)
- Escalación horizontal o vertical de privilegios

### ¿Cómo prevenirlo?

- Implementar autorización en backend (nunca confiar en el frontend)
- Aplicar principio de mínimo privilegio
- Validar ownership de recursos
- Centralizar la lógica de autorización
- Pruebas automatizadas de permisos
- Revisiones periódicas de roles y scopes

## Security Misconfiguration
### Descripción
Errores en configuraciones de infraestructura, frameworks o nube que exponen la aplicación.
Incluye:

- Debug habilitado en producción
- CORS abierto
- Headers de seguridad ausentes
- Permisos excesivos en cloud

### ¿Cómo prevenirlo?
- Baselines de configuración seguras
- Infraestructura como código (IaC) (Nuestro estandar dentro de Scisa es Terraform)
- Hardening de servidores
- Escaneos automatizados de configuración
- Deshabilitar funcionalidades innecesarias
- Revisiones periódicas de entornos productivos

## Software Supply Chain Failures
### Descripción
Riesgos derivados del uso de dependencias, librerías, paquetes o pipelines comprometidos.

Incluye:
- Dependencias con CVEs conocidos
- Ataques a paquetes NPM/NuGet/PyPI
- CI/CD comprometido

### ¿Cómo prevenirlo?
- Escaneo de dependencias (SCA) (El estandar dentro de Scisa es Checkmarks, por lo que es importante revisar constantemente el reporte de vulnerabilidades que genera esta herramienta)
- Actualizaciones periódicas obligatorias
- Bloqueo de versiones críticas
- Firmar artefactos
- Control de acceso a pipelines
- Uso de repositorios internos de paquetes

## Cryptographic Failures
### Descripción
Uso incorrecto o débil de mecanismos criptográficos que exponen información sensible.
Incluye:
- Contraseñas sin hash
- Algoritmos obsoletos
- Claves expuestas en código (Revisar guia sobre [variables y secretos](./09.2.variables_secrets.md))

### ¿Cómo prevenirlo?
- Forzar HTTPS
- Usar algoritmos modernos (AES-256, RSA-2048+, bcrypt, Argon2)
- No inventar criptografía personalizada
- Gestión segura de secretos (Secret Manager / Vault)
- Rotación periódica de claves

## Injection
### Descripción
Cuando datos externos se interpretan como comandos o consultas ejecutables.
Incluye:

- SQL Injection
- NoSQL Injection
- Command Injection
- LDAP Injection

### ¿Cómo prevenirlo?
- Consultas parametrizadas
- Uso correcto de ORM (Estandar dentro de Scisa es Entity Framework, revisar guias sorbe [buenas prácticas](/docs/programming/0.4.good_practices/04.2.SOLID.md))
- Validación estricta de inputs
- Sanitización de datos
- Principio de separación de comandos y datos

## Insecure Design
### Descripción
Problemas de seguridad originados en decisiones de arquitectura o diseño, no necesariamente en bugs.
Incluye:
- Flujos de negocio manipulables
- Falta de rate limiting
- No considerar abuso del sistema

### ¿Cómo prevenirlo?
- Threat Modeling desde la fase de diseño
- Security by Design
- Revisiones arquitectónicas
- Definir controles compensatorios
- Validación de casos de abuso

## Authentication Failures
### Descripción
Errores en los mecanismos de autenticación y gestión de sesiones.

Incluye:
- Tokens sin expiración
- Sesiones mal invalidadas
- Falta de MFA
- Fuerza bruta sin protección

### ¿Cómo prevenirlo?
- Implementar MFA
- Expiración y rotación de tokens
- Protección contra fuerza bruta
- Cookies seguras (HttpOnly + Secure)
- No almacenar contraseñas en texto plano

## Software or Data Integrity Failures
### Descripción
Manipulación de código o datos sin verificación de integridad.

Incluye:
- Deserialización insegura
- Artefactos sin firma
- Actualizaciones sin validación

### ¿Cómo prevenirlo?
- Firmar código y artefactos
- Validar integridad en CI/CD
- Restringir ejecución de código dinámico
- [Control de acceso a repositorios](./09.1.security_guidelines.md)

## Security Logging and Alerting Failures
### Descripción
Falta de registros y monitoreo que impide detectar ataques oportunamente.

Incluye:
- No registrar intentos fallidos de login
- No generar alertas de comportamiento anómalo
- Logs incompletos o inseguros

### ¿Cómo prevenirlo?
- Centralizar logs (puedes revisar la guia de logs en [09.3.logging.md](/docs/programming/0.4.good_practices/04.5.logs.md))
- Alertas automáticas
- Retención adecuada de logs
- Monitoreo activo 24/7
- Proteger los logs contra manipulación

## Mishandling of Exceptional Conditions
### Descripción
Manejo incorrecto de errores y excepciones que expone información sensible o genera fallas críticas.

Incluye:
- Stack traces visibles en producción
- Mensajes de error detallados
- Fallos que dejan el sistema en estado inconsistente

### ¿Cómo prevenirlo?
- Manejo global de excepciones
- Mensajes de error genéricos al usuario
- Logs detallados solo internamente
- Circuit breakers y mecanismos de resiliencia
- Pruebas de fallos controlados (chaos testing)
