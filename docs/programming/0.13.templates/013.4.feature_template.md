# Template para documentación basada en Features

Una buena forma de documentar features con flujo de usuario es usar una estructura que combine:

- Contexto de negocio
- Actores
- Precondiciones
- Flujo principal
- Escenarios alternos / errores
- Reglas del sistema
- Interacciones técnicas

Esto hace que la documentación sirva tanto para desarrolladores, QA y revisores de PR.

## Template

```md
# Feature: <Nombre de la feature>  (Requerido)

## 1. Descripción (Requerido)

Breve explicación del objetivo de la funcionalidad y el problema que resuelve.

---

## 2. Actores (Requerido)

Lista de actores involucrados en el flujo.

Ejemplo:

- Usuario
- Administrador
- Sistema
- Servicio externo

---

## 3. Contexto de Negocio (Opcional)

Explica **por qué existe la funcionalidad** o qué problema de negocio resuelve.

---

## 4. Precondiciones (Opcional)

Condiciones que deben cumplirse antes de iniciar el flujo.

Ejemplo:

- El usuario debe existir en el sistema
- El usuario debe tener credenciales válidas
- La feature debe estar habilitada

---

## 5. Flujo Principal (Happy Path) (Requerido)

Describe el flujo esperado cuando todo funciona correctamente.

### Paso 1
Descripción del primer paso.

### Paso 2
Descripción del siguiente paso.

### Paso 3
Resultado esperado.

---

## 6. Escenarios Alternativos (Opcional)

Describe variaciones del flujo principal.

### Escenario A
Condición que cambia el flujo principal.

Flujo alternativo.

### Escenario B
Otro flujo alternativo.

---

## 7. Manejo de Errores (Opcional)

| Error | Código | Comportamiento |
|------|------|------|
| Descripción del error | ERROR_CODE | Mensaje o comportamiento del sistema |

---

## 8. Reglas de Negocio (Opcional)

Restricciones o validaciones que el sistema debe cumplir.

Ejemplo:

- El PIN debe ser numérico
- El valor mínimo debe ser X
- No se permite continuar si la validación falla

---

## 9. Interacciones Técnicas (Opcional)

### Servicios utilizados

| Servicio | Método | Descripción |
|------|------|------|
| ServiceName | MethodName | Descripción |

### Datos generados o utilizados

| Campo | Descripción |
|------|------|
| field_name | descripción |

---

## 10. Permisos y Seguridad (Opcional)

Si la funcionalidad depende de permisos.

| Permiso | Descripción |
|------|------|
| permission_name | Qué permite hacer |

---

## 11. UI / Comportamiento esperado (Opcional)

Descripción del comportamiento visual esperado.

Ejemplo:

- Pantallas involucradas
- Inputs
- Botones
- Mensajes

---

## 12. Eventos o Logs (Opcional)

Eventos que deben registrarse en logs o auditoría.

| Evento | Descripción |
|------|------|
| LOGIN_SUCCESS | Usuario inició sesión |

---

## 13. Dependencias (Opcional)

Servicios o sistemas externos necesarios.

Ejemplo:

- Servicio de autenticación
- Base de datos
- API externa

---

## 14. Notas Técnicas (Opcional)

Consideraciones para implementación.

Ejemplo:

- El PIN puede recibirse como string pero debe validarse como numérico
- El SecretKey solo debe enviarse en el flujo inicial
```

## Ejemplo 

```md
# Feature: Autenticación con Two Factor Authentication (2FA)

## 1. Descripción

Permite agregar una segunda capa de seguridad al proceso de login mediante aplicaciones como Google Authenticator.

---

## 2. Actores

- Usuario
- Administrador
- Sistema PLD

---

## 3. Precondiciones

- El usuario debe tener credenciales válidas
- El usuario debe tener 2FA habilitado

---

## 4. Flujo Principal (Happy Path)

### Paso 1 – Login
El usuario ingresa sus credenciales.

### Paso 2 – Verificación de 2FA
El sistema detecta que el usuario tiene 2FA habilitado.

### Paso 3 – Solicitud de PIN
Se muestra una pantalla solicitando el código generado por la app.

### Paso 4 – Validación
El sistema valida el PIN.

### Paso 5 – Login exitoso
Se crea la sesión del usuario y se obtiene su perfil.

---

## 5. Escenarios Alternativos

### Usuario no emparejado con 2FA

Si el usuario tiene 2FA habilitado pero no está emparejado (`ERROR_CODE_USER_NON_PAIRED_TWO_FACTOR_AUTH`):

1. Se genera una clave secreta.
2. Se genera un código QR.
3. El usuario escanea el QR.
4. El usuario ingresa el código generado por su app.

---

### Usuario pierde acceso a 2FA

Un administrador puede reiniciar el emparejamiento.

El usuario deberá escanear nuevamente el QR.

---

## 6. Manejo de Errores

| Error | Código | Comportamiento |
|------|------|------|
| Usuario no emparejado | ERROR_CODE_USER_NON_PAIRED_TWO_FACTOR_AUTH | Se muestra pantalla de emparejamiento |
| Código incorrecto | INVALID_PIN | Mostrar mensaje de error |

---

## 7. Reglas de Negocio

- El PIN debe ser numérico
- Debe tener al menos 4 dígitos
- No se permite continuar si el PIN es inválido

---

## 8. Interacciones Técnicas

### Servicios utilizados

| Servicio | Método | Descripción |
|------|------|------|
| LoginService | GenerateSecretKey | Genera clave secreta |
| LoginService | ValidateTwoFactorPIN | Valida el PIN |

---

## 9. Permisos y Seguridad

| Permiso | Descripción |
|------|------|
| ResetTwoFactorAuth | Permite reiniciar emparejamiento 2FA |

---

## 10. UI / Comportamiento esperado

Pantalla de validación:

- Input numérico
- Botón confirmar
- Botón regresar al login

Pantalla de emparejamiento:

- QR
- Input para código
- Botón confirmar

---

## 11. Notas Técnicas

El PIN puede recibirse como string pero debe validarse como numérico.
```
