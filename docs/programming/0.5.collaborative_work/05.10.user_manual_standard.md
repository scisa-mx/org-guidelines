# Guía Organizacional — Manuales de Usuario como Parte del Proceso de Desarrollo

## 1. ¿A qué aplica esta guía?

Esta guía aplica a cualquier feature o módulo que sea entregado al usuario final como funcionalidad operativa. Específicamente: al momento en que un desarrollador o equipo de producto declara un feature como listo para producción.

El responsable del ticket (desarrollador o PM, según el tipo de cambio) es responsable de:

- Verificar antes de cerrar el ticket que lo que el usuario puede hacer en producción y lo que el manual describe son consistentes.

Esta guía **no aplica** a cambios internos de infraestructura, refactors que no alteran comportamiento observable para el usuario, o ajustes de configuración sin impacto en la interfaz.

---

## 2. ¿Por qué documentamos con manuales de usuario?

El código describe *cómo* funciona el sistema. La documentación BDD describe *qué* debe hacer según las reglas de negocio. El manual de usuario describe *cómo opera una persona real dentro del sistema*. Los tres son complementarios y ninguno reemplaza a los otros.

Sin manuales de usuario, el equipo de soporte responde las mismas preguntas de forma repetida, los usuarios nuevos requieren capacitación presencial, y el conocimiento operativo del sistema vive únicamente en las personas que lo construyeron o que llevan más tiempo usándolo.

Con manuales de usuario estructurados bajo estándares reconocidos (IEEE 1063 / ISO/IEC/IEEE 26514:2022), el comportamiento observable del sistema queda descrito desde la perspectiva de quien lo usa, en un lenguaje accesible, versionado junto al código y actualizable como cualquier otro artefacto del repositorio.

Los beneficios concretos son:

- **Autogestión del usuario.** Un manual bien redactado permite que el usuario resuelva sus dudas operativas sin necesitar al equipo de desarrollo.
- **Reducción de carga en soporte.** Las preguntas frecuentes documentadas en el manual eliminan tickets de soporte repetitivos y predecibles.
- **Base de onboarding.** Un nuevo usuario del sistema —interno o externo— puede volverse operativo leyendo el manual sin requerir sesión de capacitación.
- **Trazabilidad funcional.** Cada PR que introduce o modifica comportamiento visible para el usuario debe actualizarse el manual. El manual es evidencia de lo que se entregó.
- **Referencia para QA.** Los procedimientos documentados son el punto de partida para casos de prueba de aceptación.

---

## 3. A dónde queremos llegar

El objetivo es que el manual de usuario sea el contrato operativo de cada feature: lo que describe el manual es lo que el usuario puede hacer, sin excepción.

Cuando lleguemos ahí:

- Cualquier persona nueva puede aprender a usar el sistema de forma autónoma, sin necesitar a alguien del equipo de desarrollo.
- El equipo de soporte tiene una referencia oficial a la que puede remitir a los usuarios antes de escalar al equipo técnico.
- QA puede diseñar casos de prueba directamente desde los procedimientos documentados, sin necesitar al desarrollador.
- Las regresiones funcionales son detectables: si un cambio de código altera un flujo documentado, el manual debe actualizarse — y ese cambio es visible en el PR.
- La documentación puede entregarse en múltiples formatos (WebHelp, PDF, Markdown) según el canal que necesite cada audiencia.

---

## 4. Riesgos de confiar en el manual en lugar de probarlo

El manual de usuario no es el sistema. Es una descripción del sistema desde la perspectiva operativa. Si se gestiona mal, puede convertirse en una fuente de desinformación y erosionar la confianza del usuario.

### Riesgo principal: el manual se desactualiza

Si el sistema cambia pero el manual no, el usuario seguirá instrucciones que ya no corresponden a lo que ve en pantalla.

**Cómo minimizarlo:**

- **El cierre del ticket es el guardián.** Ningún feature puede declararse terminado sin haber revisado y actualizado el manual correspondiente.
- **Los nombres de elementos de la interfaz son exactos.** Si un botón se renombra en el sistema, debe renombrarse en cada lugar del manual donde aparezca. La precisión semántica no es opcional.
- **Las capturas de pantalla deben reflejar el estado actual.** Una imagen desactualizada es más confusa que no tener imagen.

### Riesgo secundario: documentar comportamiento futuro como presente

Escribir procedimientos de funcionalidades que aún no están implementadas, como si ya existieran en el sistema.

**Cómo minimizarlo:**

- El manual solo describe lo que está disponible en la versión activa del sistema. El comportamiento planeado vive en el backlog, no en el manual.
- Si hay una funcionalidad en desarrollo, puede mencionarse como "próximamente disponible" con una nota explícita — nunca como un procedimiento operativo completo.

### Riesgo terciario: el procedimiento es correcto pero impreciso

Un procedimiento puede estar técnicamente correcto pero ser tan ambiguo que el usuario no pueda seguirlo sin ayuda.

**Cómo minimizarlo:**

- Cada paso debe ser lo suficientemente específico para que un usuario sin conocimiento previo pueda reproducirlo. El criterio de validación es simple: si alguien ajeno al equipo puede completar el flujo leyendo el manual, el manual es suficientemente preciso.
- Incluir valores concretos, nombres exactos de campos y botones, y describir el resultado esperado después de cada acción.

## 5. Estándar de redacción

El manual no es un documento informal. Sigue un lenguaje controlado basado en IEEE 1063 e ISO/IEC/IEEE 26514:2022. Estas reglas no son sugerencias — son el estándar del equipo.

### Reglas de estilo

| Regla | Correcto | Incorrecto |
|---|---|---|
| Voz activa | "Haga clic en **Guardar**" | "El botón Guardar debe ser presionado" |
| Segunda persona | "Seleccione el archivo" | "El usuario selecciona el archivo" |
| Imperativo para pasos | "Ingrese su correo electrónico" | "Hay que ingresar el correo" |
| Precisión semántica | Siempre "**Enviar solicitud**" | "Enviar" / "Mandar" / "Confirmar" para el mismo botón |
| Frases cortas | Máximo 25–30 palabras por oración | Oraciones con múltiples cláusulas subordinadas encadenadas |

### Precisión semántica — regla crítica

Cada elemento de la interfaz tiene un nombre único y ese nombre se usa de forma idéntica en todo el manual. Si el botón en el sistema dice **"Guardar cambios"**, el manual dice "Guardar cambios" en cada mención — nunca "Guardar", "Aplicar" ni ningún sinónimo.

Si el nombre de un elemento cambia en el sistema, cambia en todo el manual. Esta consistencia es lo que hace que el manual sea confiable.

### Formato de advertencias y notas

```
NOTA: Información adicional útil, no bloqueante para completar la tarea.
IMPORTANTE: Condición que el usuario debe cumplir para evitar errores.
RESTRICCIÓN: Acción que el sistema no permite o límite operativo conocido.
RESULTADO ESPERADO: Qué debe ver el usuario después de ejecutar el paso.
```

## 6. Estructura obligatoria del manual (IEEE 1063)

Todo manual debe seguir esta estructura. Las secciones pueden ajustarse según el tipo de manual, pero ninguna puede omitirse si el contenido aplica.

```
1. DATOS DE IDENTIFICACIÓN
   · Nombre del sistema o módulo documentado
   · Tipo de manual
   · Versión del documento
   · Fecha de emisión
   · Audiencia objetivo (Usuario final / Operador / Administrador)
   · Área responsable

2. INTRODUCCIÓN Y ALCANCE
   · Qué hace el sistema o módulo
   · A quién se dirige este manual
   · Qué incluye y qué excluye esta documentación
   · Cómo navegar el documento

3. PRERREQUISITOS
   · Accesos y permisos necesarios
   · Software o conectividad requerida
   · Rol dentro del sistema

4. PROCEDIMIENTOS (repetir por flujo o pantalla)
   · Nombre del procedimiento
   · Objetivo
   · Condición previa
   · Pasos secuenciales numerados con resultado esperado

5. TROUBLESHOOTING
   · Tabla de errores: mensaje exacto, causa probable, acción correctiva
   · Árbol de decisión si hay múltiples rutas de diagnóstico

6. GLOSARIO
   · Términos técnicos con definición precisa, sin sinónimos, en orden alfabético

7. REFERENCIAS Y SOPORTE
   · Documentos relacionados
   · Canal de soporte y tiempo de respuesta esperado
```

## 7. Checklist de consistencia antes de cerrar un ticket

Antes de abrir el PR, el responsable debe verificar que el manual y el sistema son consistentes. Esta revisión no la hace ningún pipeline automatizado — la hace la persona.

El objetivo es un **match del 100%** entre lo que describe el manual y lo que el usuario puede hacer en el sistema.

**Checklist:**

- [ ] Cada flujo nuevo o modificado implementado en el código tiene un procedimiento correspondiente en el manual.
- [ ] Los nombres de todos los campos, botones y mensajes en el manual coinciden exactamente con los que aparecen en la interfaz.
- [ ] Las capturas de pantalla reflejan el estado actual del sistema (no staging, no versiones anteriores).
- [ ] Los casos de error identificados durante el desarrollo están en la tabla de troubleshooting.
- [ ] No hay procedimientos documentados que describan funcionalidad que aún no está disponible en producción.

Si hay algo que no cuadra, la respuesta no es ignorarlo. Es resolver la desalineación siguiendo la guía de la Sección 9.

---

## 8. Casos de desalineación y cómo resolverlos

Durante el desarrollo o al momento de revisar el ticket, pueden aparecer tres tipos de desalineación entre el manual y el sistema.

---

### Caso A: El sistema tiene flujos que el manual no documenta

**Qué pasó:** Durante el desarrollo se implementaron comportamientos o pantallas que no estaban en el análisis inicial.

**Qué hacer:** Agregar los procedimientos faltantes al manual. La implementación descubrió comportamiento real que el usuario va a encontrar — y necesita documentación.

**Ejemplo:**

> El manual documenta cómo crear un registro. Durante el desarrollo se implementó también la posibilidad de duplicar un registro existente. Ese flujo debe agregarse al manual antes de cerrar el ticket.

---

### Caso B: El manual documenta flujos que el sistema no implementa

**Qué pasó:** El análisis fue más optimista que la implementación, o parte del alcance se recortó.

**Qué hacer:** Dos opciones según el contexto:

1. **Si el flujo faltante estaba dentro del alcance del ticket:** Implementarlo. Si no hay tiempo, crear un nuevo ticket para esa funcionalidad y actualizar el manual para reflejar únicamente lo que está disponible.
2. **Si el flujo quedó fuera del alcance conscientemente:** Eliminar o marcar como "próximamente disponible" en el manual. El comportamiento pendiente va al backlog, no a la documentación operativa.

**Lo que no se puede hacer:** Dejar el manual describiendo un flujo que el usuario no puede ejecutar en el sistema actual.

---

### Caso C: El manual y el sistema se contradicen en detalles

**Qué pasó:** El nombre de un campo cambió, el orden de los pasos se modificó, o aparecen mensajes de error distintos a los documentados.

**Qué hacer:** Actualizar el manual para que coincida exactamente con el estado actual del sistema. Si el cambio es significativo y afecta a usuarios que ya leyeron la versión anterior, notificarlo en el changelog del documento.

**Ejemplo:**

> El manual dice que el campo se llama "Correo corporativo" pero en la UI actual dice "Correo institucional". Además, el botón de confirmación fue renombrado de "Enviar" a "Enviar solicitud". Ambos cambios deben reflejarse en el manual antes de cerrar el ticket.

---

## 9. Consistencia sobre completitud

El objetivo de esta guía no es tener manuales exhaustivos. Es tener manuales **consistentes con el sistema**.

Un manual con tres procedimientos bien documentados, con capturas actualizadas y nombres exactos, es mejor que uno con diez procedimientos donde la mitad describe pantallas que ya no existen.

**Hay espacio para el crecimiento.** Los manuales pueden expandirse, los procedimientos pueden refinarse, el alcance puede aumentar con el tiempo. Lo que no es aceptable es que el sistema y el manual digan cosas distintas al momento de cerrar un ticket.

Cuando el equipo interioriza este principio, el manual deja de ser una carga y se convierte en una herramienta de trabajo: el primer lugar al que alguien va cuando tiene una duda sobre cómo usar el sistema, y el último lugar que se revisa antes de declarar un ticket como terminado.
