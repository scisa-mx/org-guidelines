# Definition of Done

Un ticket no se considera terminado hasta que cumple **los dos niveles** de este DoD. Ninguno reemplaza al otro.

---

## Nivel 1 — Criterios de aceptación del ticket

Todos los criterios de aceptación definidos en el ticket deben estar verificados con evidencia. Si no hay criterios de aceptación en el ticket, el ticket no estaba listo para desarrollo (ver DoR `05.3.DoR.md`).

- [ ] Todos los AC del ticket verificados
- [ ] Evidencia registrada en el ticket: capturas, logs, resultados de Postman o equivalente
- [ ] Si se descubrieron casos borde no contemplados en los AC originales: documentados en el ticket y confirmados por el Analista/PO
- [ ] Si se definieron benchmarks en la propuesta: ejecutados y resultados adjuntos con evidencia

## Nivel 2 — Criterios técnicos universales

Aplican a todo PR sin excepción, sin importar el tipo de cambio.

- [ ] Code review completado por al menos un par; todos los comentarios resueltos
- [ ] CHANGELOG actualizado con la entrada correspondiente
- [ ] BDD doc en `/docs/features/<nombre>/` creado o actualizado
- [ ] Todos los tests existentes pasan; no se introdujeron regresiones
- [ ] El código cumple los estándares del equipo (linter, naming, estructura)
- [ ] No se introdujo deuda técnica nueva; si fue inevitable, está registrada en el backlog

---

## Qué NO es "Done"

- "Funciona en mi máquina" no es evidencia.
- Un ticket con AC parcialmente cubiertos no está Done — está parcialmente hecho.
- Si el Analista/PO no confirmó los casos borde nuevos, no está Done.

---

## Artefactos de entrega final (nivel proyecto)

Al cerrar un proyecto o release completo, además del DoD por ticket, el **Gateway de entrega** requiere:

- Diagrama de arquitectura actualizado en el repo
- Manual de usuario actualizado y consistente con el sistema entregado
- Benchmarks ejecutados con evidencia (si fueron definidos en la propuesta)
- CHANGELOG al día para la versión que se entrega

Ver `05.12.delivery_gateway.md`.
