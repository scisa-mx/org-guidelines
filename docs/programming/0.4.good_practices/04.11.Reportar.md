# Si obsercas algo, reportalo.

## ¿Qué NO significa “si observas algo”?

No significa:

- Arreglar todo lo que veas en producción
- Meter cambios extra en un PR ajeno
- Reescribir módulos completos “ya que estamos aquí”

Significa: No ignorar deuda, errores o riesgos evidentes cuando los detectas.

## ¿Qué cosas sí deben reportarse?

### Bugs evidentes

NullReference esperando explotar
Validaciones faltantes
Manejo incorrecto de errores

### Problemas de diseño claros

Clases con demasiadas responsabilidades
DTOs mutables cruzando capas
Lógica de negocio en controllers

### Riesgos técnicos

Código no thread-safe
Uso incorrecto de async/await
Recursos no liberados

### Violaciones a lineamientos

Estándares de logging
Inmutabilidad
Clean Architecture

## Reportar ≠ Corregir inmediatamente

Este punto es clave y suele romper el flujo de los equipos.
Cuando alguien detecta un problema, la reacción instintiva suele ser:

- Detectar algo
- “Arreglarlo de pasada”
- Meterlo en el PR actual

Esto genera:

- PRs enormes
- Revisiones lentas
- Conflictos innecesarios
- Pérdida de contexto
- Se pierde el flujo de trabajo
- Se atrasa la corrección real
- Se atrasan los tickets

## El flujo correcto: reporta → evalúa → corrige en su branch

### Paso 1: Reportar
Levanta la mano, reportalo con tu equipo y creen un ticket si es necesario. Si esto esta dentro de un PR, genera un comentario dentro de él.

### Paso 2: Evaluar impacto
Antes de tocar código, pregúntate:

- ¿Rompe funcionalidad?
- ¿Es riesgo inmediato o deuda técnica?
- ¿Es quick fix o refactor grande?

Esto por si no es un único problema, sino varios. Hay que priorizarlos y atacarlos de forma separada.

### Paso 3: Corregir en su branch

Si se decide corregir:

- Crear branch específico
- Scope claro
- Cambio enfocado

## ¿Cuándo SÍ corregir de inmediato?
Hay excepciones claras:

Casos críticos: 

- Bugs de seguridad
- Pérdida de datos
- Caídas en producción

## ¿Cuándo NO corregir tú mismo?
No te corresponde si:

- El cambio afecta dominio crítico sin contexto
- Es un refactor grande
- Requiere decisiones de arquitectura
