# 2.4 Estructura de Proyectos y Solución

La estructura de un repositorio no es opcional ni cosmética: es la primera documentación que lee cualquier desarrollador que entra al proyecto. Una estructura consistente reduce el tiempo de onboarding, hace predecible dónde buscar cada cosa y facilita el mantenimiento a largo plazo.

---

## Arquitectura por capas

Todos los aplicativos de SCISA siguen una arquitectura por capas. Cada capa tiene una responsabilidad única y no debe mezclar responsabilidades de otra.

| Capa | Proyecto | Responsabilidad |
|---|---|---|
| **Presentación** | `Presentation.WPF` / `MVC` / `API` | UI, controladores, endpoints. Solo orquesta, no contiene lógica de negocio. |
| **Servicios** | `Services` | Expone operaciones hacia afuera (WCF, REST). Coordina entre Business y Data. |
| **Contratos** | `Services.Contracts` | Interfaces y DTOs que definen el contrato público del servicio. Sin implementación. |
| **Negocio** | `Business.Components` | Lógica de negocio pura. No conoce la base de datos ni la UI. |
| **Entidades** | `Business.Entities` | Modelos de dominio, DTOs, enumeraciones. Sin lógica compleja. |
| **Datos** | `Data` | Acceso a base de datos (repositorios, queries, migraciones). |
| **Jobs** | `Jobs` | Procesos programados o en background. Consume `Business`, nunca `Data` directamente. |
| **Colas** | `AMQ` / `Queues` | Integración con sistemas de mensajería (ActiveMQ, RabbitMQ, etc.). |
| **Aplicación** | `Application` | Punto de entrada y composición del contenedor de dependencias. |
| **I18n** | `I18n` | Recursos de internacionalización y localización. |

> La dependencia siempre va hacia abajo: `Presentation → Services → Business → Data`. Nunca al revés.

## 2.4.3 Nuevos aplicativos

Para cualquier aplicativo nuevo, la estructura mínima es:

```
docs/
    adr/
    features/
src/
    Scisa.<Namespace>
    <Modulo>/
        Scisa.<Modulo>.<Namespace>
test/
    Scisa.<Namespace>.Tests.Integration
    Scisa.<Namespace>.Tests.Unit
README.md
CHANGELOG.md
.editorconfig
.gitignore
```

### Cómo elegir el `<Namespace>`

El namespace identifica la capa o responsabilidad del proyecto dentro de la solución. Sigue el patrón:

```
Scisa.<Producto>.<Capa>
```

Ejemplos:

| Proyecto | Qué es |
|---|---|
| `Scisa.Pagos.Business.Components` | Lógica de negocio del módulo de pagos |
| `Scisa.Pagos.Business.Entities` | Entidades y DTOs del módulo de pagos |
| `Scisa.Pagos.Data` | Acceso a datos del módulo de pagos |
| `Scisa.Pagos.Services` | Capa de servicios expuestos |
| `Scisa.Pagos.Services.Contracts` | Interfaces y contratos del servicio |
| `Scisa.Pagos.API` | Endpoints REST del módulo |
| `Scisa.Pagos.Tests.Unit` | Pruebas unitarias |
| `Scisa.Pagos.Tests.Integration` | Pruebas de integración |

> Usa PascalCase. Sin abreviaciones, sin guiones. El nombre debe ser descriptivo del dominio, no de la tecnología.

## Qué va en cada carpeta raíz

### `docs/`

Toda la documentación técnica del repositorio. No es un repositorio de Word o PDFs: es Markdown vivo que evoluciona junto al código.

```
docs/
    adr/                        ← Architecture Decision Records
        0001-eleccion-orm.md
        0002-autenticacion-jwt.md
    features/
        <nombre-del-feature>/
            <nombre-del-feature>.md     ← contexto funcional y escenarios BDD
```

- **`adr/`** — Decisiones arquitectónicas relevantes. Se crea un ADR cada vez que se elige una tecnología, patrón o enfoque que no es obvio. Ver [documentación mínima requerida](../0.5.collaborative_work/05.6.min_docs.md).
- **`features/`** — Un directorio por feature, con su `.md` describiendo contexto, actores y escenarios Gherkin.

### `src/`

Todo el código fuente de producción. Nunca mezcles código de pruebas aquí.

Organiza los proyectos dentro de `src/` por módulo o dominio funcional cuando el aplicativo tenga más de una responsabilidad clara:

```
src/
    Scisa.Pagos.Business.Components/
    Scisa.Pagos.Business.Entities/
    Scisa.Pagos.Data/
    Scisa.Pagos.Services/
    Scisa.Pagos.Services.Contracts/
    Scisa.Pagos.API/
```

### `test/`

Proyectos de prueba separados del código de producción. Nombra los proyectos espejando el proyecto que prueban, con el sufijo `.Tests.Unit` o `.Tests.Integration`.

```
test/
    Scisa.Pagos.Tests.Unit/
    Scisa.Pagos.Tests.Integration/
```

- **Unit** — pruebas sin dependencias externas. Rápidas, sin base de datos, sin red.
- **Integration** — pruebas que validan el comportamiento end-to-end contra dependencias reales (BD, servicios externos).

## Archivos raíz obligatorios

Todo repositorio nuevo debe incluir los siguientes archivos desde el primer commit:

| Archivo | Para qué sirve |
|---|---|
| `README.md` | Explica qué hace el proyecto, cómo levantarlo localmente y dónde encontrar la documentación principal. |
| `CHANGELOG.md` | Historial de cambios significativos organizado por versión. |
| `.editorconfig` | Configuración de formato de código compartida entre todos los IDEs. Usa la [copia oficial del org](https://github.com/scisa-mx/org-guidelines/blob/main/.editorconfig). |
| `.gitignore` | Archivos que Git debe ignorar: binarios, secretos, carpetas de IDE, artefactos de build. |
| `adr` | Explica y documenta las decisiones arquitectónicas relevantes. |
| `features` | Documenta el contexto funcional y los escenarios de cada feature. |

### `README.md` mínimo

Un README cumple su función si responde estas tres preguntas:

1. ¿Qué hace este proyecto?
2. ¿Cómo lo levanto en mi máquina? (variables de entorno, comandos de setup)
3. ¿Dónde están los flujos principales y la documentación?

No tiene que ser largo. Tiene que ser útil.

### `CHANGELOG.md` mínimo

Sigue el formato [Keep a Changelog](https://keepachangelog.com/es/1.0.0/). Cada release debe tener su sección con los cambios agrupados por tipo: `Added`, `Changed`, `Fixed`, `Removed`.

## Checklist para un repositorio nuevo

Antes de hacer el primer PR al repositorio nuevo, verifica:

- [ ] La solución sigue el patrón de nombres `Scisa.<Producto>.<Capa>`
- [ ] Existe `src/` con los proyectos separados por capa
- [ ] Existe `test/` con al menos el proyecto de pruebas de integración
- [ ] Existe `docs/` con las carpetas `adr/` y `features/`
- [ ] El `README.md` responde las tres preguntas mínimas
- [ ] Hay un `CHANGELOG.md` con la entrada de la versión inicial
- [ ] El `.editorconfig` es una copia de la configuración oficial
- [ ] El `.gitignore` cubre binarios, secretos y artefactos de build
- [ ] El aplicativo no tiene dependencias circulares entre capas
- [ ] Los proyectos de prueba están en `test/`, no mezclados en `src/`
