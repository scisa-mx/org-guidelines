# Checklist de Propuesta

Este checklist aplica antes de presentar una propuesta al cliente. Su propósito es detectar ambigüedad antes de que llegue a desarrollo — no es una lista de artefactos a producir, es una lista de preguntas que exponen huecos.

**La propuesta no se presenta sin haber pasado este checklist.**

La propuesta aprobada y firmada por el cliente es el gate de inicio de desarrollo. Iniciar antes es un riesgo asumido conscientemente, no una práctica estándar.

---

## Claridad funcional

- [ ] ¿Cada proceso está narrado paso a paso, sin saltos que requieran interpretación?
- [ ] ¿Hay mockups, wireframes o diagramas que muestren cómo se verá lo que se propone?
- [ ] ¿Los escenarios BDD cubren el flujo feliz *y* los casos de error relevantes?
- [ ] ¿Hay algún término que el cliente y el equipo puedan entender de forma distinta? ¿Está definido explícitamente en la propuesta?

## Alcance

- [ ] ¿Qué está explícitamente *excluido* del alcance? (El "qué NO se hace" es tan importante como el "qué sí")
- [ ] ¿Las dependencias con sistemas existentes del cliente o terceros están identificadas?
- [ ] ¿El cliente revisó y validó que los escenarios BDD describen exactamente lo que quiere?

## Requerimientos técnicos

- [ ] ¿Hay requerimientos no funcionales definidos (performance, disponibilidad, seguridad, escalabilidad)?
- [ ] ¿Se requieren benchmarks? Si sí: ¿qué se mide, contra qué baseline y cuál es el umbral de aceptación?
- [ ] ¿Las decisiones de arquitectura significativas que ya se pueden anticipar están documentadas como ADRs?

## Entregables comprometidos

- [ ] ¿El diagrama de arquitectura de la solución está incluido o referenciado?
- [ ] ¿El manual de usuario es parte del entregable?
- [ ] ¿El cliente sabe qué recibirá: código, documentación, manuales, benchmarks?

## Relación con otros documentos

| Artefacto | Dónde vive | Cuándo se genera |
|---|---|---|
| Documento de propuesta completo (formato presentable) | Google Drive / carpeta del proyecto | Al armar la propuesta |
| Escenarios BDD de la propuesta | Drive (propuesta) → Repo por PR al implementar cada feature | Al proponer y al desarrollar |
| ADRs de decisiones arquitectónicas | Repo `/docs/adr/` | Antes de iniciar desarrollo |
| Diagrama de arquitectura | Repo (Mermaid) | Como entregable de cierre |
| Manual de usuario | Repo Docusaurus independiente | Como entregable de cierre |
