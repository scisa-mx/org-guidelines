# Pair programing y revisiones colaborativas 

Es importante establecer cómo y cuándo utilizar Pair Programming y Revisiones Colaborativas para:

- Mejorar la calidad del código
- Reducir errores tempranos
- Compartir conocimiento
- Evitar silos técnicos
- Alinear decisiones técnicas

## ¿Qué es el Pair programming?
Práctica donde dos desarrolladores trabajan juntos sobre el mismo código:

- Uno escribe (Driver), uno revisa y piensa (Navigator) y los roles rotan.

## Cuándo usar Pair Programming

Pair Programming SÍ se recomienda cuando:

- Se trabaja en lógica compleja o crítica
- Se toman decisiones de arquitectura
- Se refactoriza código sensible
- Se onboardea a alguien nuevo
- Se detecta riesgo alto de defectos

Si el cambio es difícil de explicar, probablemente vale la pena hacerlo en pair.

## Cuándo NO usar Pair Programming

Pair Programming NO es obligatorio cuando:

- El cambio es mecánico o repetitivo
- Es un fix trivial
- El contexto ya es ampliamente conocido
- El costo supera el beneficio

## Cuándo usar Revisiones Colaborativas

Las revisiones colaborativas:

- Son obligatorias para todo código
- Funcionan como red de seguridad
- Complementan al pair programming

| Práctica              | Momento      | Objetivo                     |
| --------------------- | ------------ | ---------------------------- |
| Pair Programming      | Antes del PR | Diseñar y prevenir errores   |
| Revisión Colaborativa | En el PR     | Validar y compartir contexto |

## Buenas prácticas


- **Sesiones cortas (60–90 min)**: Mantener alta la concentración.
- **Objetivo claro antes de empezar**: Definir qué se busca lograr.
- **Rotar roles regularmente**: Cambiar Driver/Navigator cada 20-30 min.
- **Documentar decisiones clave**: Anotar acuerdos y razones.
- **Mantener foco técnico**: Evitar distracciones y temas no relacionados.

## DOs/DON'Ts

### DO's

- Usar pair para cambios críticos
- Combinar pair + PR
- Fomentar rotación de pares
- Priorizar aprendizaje compartido

### DON'Ts

- Usar pair como control
- Evitar PR “porque ya se hizo en pair”
- Forzar pair en todo
- Revisar superficialmente
