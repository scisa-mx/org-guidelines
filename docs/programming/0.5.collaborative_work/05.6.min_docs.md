# Guía de Documentación Activa

Escribir documentación no es llenar folios de texto que nadie leerá; es un acto de empatía con tu "yo" del futuro y con tus compañeros. La meta es que el conocimiento sea **descubrible, útil y, sobre todo, vivo.**

---

## 1. El Código como Primera Fuente de Verdad

Antes de escribir un archivo `.md`, asegúrate de que tu código no necesite traducción.

- **Nombres semánticos:** `calculate_final_price()` es mejor que `calc()`.
- **El "Por qué" vs. el "Qué":** El código dice *qué* estás haciendo. Los comentarios solo deben usarse para explicar el *porqué* — decisiones técnicas extrañas, parches por bugs externos o lógica de negocio compleja.
- **Self-Documenting:** Si una función es demasiado compleja para explicarla en una línea, divídela.

---

## 2. ¿Dónde guardar el conocimiento?

No toda la información pertenece al mismo lugar. Usa este criterio para decidir:

| Tipo de información | Destino | Ejemplo |
|---|---|---|
| **Aprendizaje general** | [Stack Overflow Teams](https://stackoverflowteams.com/c/scisa) | "Cómo configurar Docker con M1", "Solución a error X de React" |
| **Comportamiento de un feature** | Repositorio (`/docs/features/<feature>/`) | Escenarios BDD en Gherkin, contexto funcional y técnico del feature |
| **Decisiones de arquitectura** | Repositorio (`/docs/adr/`) | Por qué se eligió una tecnología, qué alternativas se descartaron |
| **Lógica de negocio / flujos** | Repositorio (`/docs` o `README.md`) | Criterios de cálculo, flujos de proceso, integraciones |
| **Arquitectura / Visuales** | Repositorio (archivos `.mermaid` o imágenes) | Diagramas de secuencia, entidad-relación, flujos de usuario |
| **Configuración local** | `README.md` | Variables de entorno, pasos de setup, comandos de arranque |

## 3. Documentación de Features — BDD

Cada feature que entre al repositorio debe estar documentado en `/docs/features/<nombre-del-feature>/`. Esta documentación vive junto al código y evoluciona con él — no es un entregable separado, es parte del PR.

### Estructura

```
/docs
└── features/
    ├── <nombre-del-feature>/
    │   ├── <nombre-del-feature>.md        ← contexto del feature
    └── ...
```

Los nombres de carpeta y archivo van en `kebab-case`, consistentes con el nombre usado en el backlog y los tickets.

### Qué debe contener el `.md`

- **Descripción general:** qué problema resuelve y cuál es su alcance.
- **Actores involucrados:** roles de usuario o sistemas externos que interactúan con el feature.
- **Precondiciones:** estado del sistema necesario para que el feature opere correctamente.
- **Dependencias:** otros features, servicios o módulos relacionados.
- **Notas técnicas relevantes:** restricciones o consideraciones de implementación.

Escenarios escritos en lenguaje Gherkin con la estructura estándar `Given / When / Then`, cubriendo flujos felices y casos borde:

```gherkin
Feature: Autenticación de usuario

  Como usuario registrado
  Quiero poder iniciar sesión con mis credenciales
  Para acceder al sistema de forma segura

  Scenario: Inicio de sesión exitoso
    Given el usuario "carlos@scisa.com.mx" está registrado en el sistema
    When el usuario ingresa su contraseña correcta
    Then el sistema redirige al dashboard principal
    And se genera un token de sesión válido

  Scenario: Credenciales incorrectas
    Given el usuario "carlos@scisa.com.mx" está registrado en el sistema
    When el usuario ingresa una contraseña incorrecta
    Then el sistema muestra un mensaje de error
    And el intento de inicio de sesión queda registrado en los logs
```

> **Regla de oro:** si el comportamiento del feature no está descrito en un `.md`, no existe desde el punto de vista de la documentación.

## 4. Architecture Decision Records (ADR)

Un ADR es un documento corto que registra una decisión arquitectónica significativa: qué se decidió, por qué, qué alternativas se consideraron y qué consecuencias tiene. No documenta cómo funciona el sistema — documenta el *razonamiento* detrás de las decisiones que le dieron forma.

Los ADRs son valiosos precisamente porque ese razonamiento suele perderse con el tiempo. Cuando el equipo cambia o el sistema crece, nadie recuerda por qué se eligió una tecnología o por qué se descartó un patrón. Un ADR deja ese contexto accesible para cualquiera que trabaje en el proyecto, ahora o en el futuro.

**Regla de inmutabilidad:** un ADR no se edita una vez aceptado. Si una decisión cambia, se crea un nuevo ADR que referencia y reemplaza al anterior. Esto preserva el historial completo del razonamiento del equipo.


Los archivos se nombran con un número secuencial de cuatro dígitos seguido de un título descriptivo en `kebab-case`. El número es permanente e irrepetible — garantiza que el orden cronológico de las decisiones sea siempre legible desde el sistema de archivos.

Puedes ver las templates en:

[Template 1 - Michael Nygard](https://github.com/joelparkerhenderson/architecture-decision-record/tree/main/locales/en/templates/decision-record-template-by-edgex)
[Template 2 - EdgeX Foundry](https://github.com/joelparkerhenderson/architecture-decision-record/tree/main/locales/en/templates/decision-record-template-by-michael-nygard)

## 5. Documentación Visual (Diagramas y Flujos)

Un diagrama bien hecho ahorra 10 reuniones de 30 minutos.

- **Diagramas como código:** siempre que puedas, usa **Mermaid.js** dentro de tus archivos Markdown. El diagrama vive en el repo y puede editarse sin herramientas externas.
- **Capturas de pantalla:** si la interfaz es compleja, una captura con anotaciones vale oro. Guárdalas en una carpeta `/assets` dentro de la carpeta de documentación correspondiente.

## 6. El Concepto de "Documentación Activa"

Para que la documentación no muera y se vuelva obsoleta, sigue estos principios:

- **Docs-as-Code:** trata la documentación como tratas el código. Si cambias una funcionalidad, el PR **no se aprueba** si no incluye la actualización de los docs relacionados. Esto está reforzado por el pipeline de validación (ver sección 7).
- **Elimina la fricción:** si algo te costó más de 15 minutos entenderlo o configurarlo, documéntalo inmediatamente. Es el momento en que el dolor está fresco y serás más preciso.
- **README minimalista:** todo repositorio debe tener un `README.md` que responda:
  1. ¿Qué hace este proyecto?
  2. ¿Cómo lo hago funcionar en mi máquina? (setup rápido)
  3. ¿Dónde están los flujos principales? (links a otros documentos)

---

## 7. Checklist de Documentación para tu PR

Antes de darle a "Submit", pregúntate:

- [ ] ¿Actualicé el `README` si agregué nuevas variables de entorno o cambié el setup?
- [ ] ¿Existe la carpeta `/docs/features/<nombre-del-feature>/` con su `.md` y su `.feature`?
- [ ] ¿Los escenarios BDD cubren tanto el flujo feliz como los casos borde relevantes?
- [ ] ¿Si esta decisión cambia la arquitectura, ya existe o se debe crear un ADR en `/docs/adr/`?
- [ ] ¿Subí los diagramas que dibujé en la pizarra o en Figma al repo en formato Mermaid o imagen?
- [ ] ¿Lo que aprendí de este bug ya está en Stack Overflow Teams para que otros no sufran lo mismo?
- [ ] ¿Si alguien nuevo entra hoy al equipo, puede entender este flujo con lo que escribí?

> **Nota:** La documentación activa es la que evoluciona. Si encuentras un doc desactualizado, corrígelo o bórralo en el momento. No lo dejes "para luego".
