# Gateway de Entrega

El Gateway de entrega es el checkpoint formal de validación previo al merge de un release a `master`. Es distinto del Gateway de producción: ocurre antes del deploy, sobre un ambiente publicado de QA, y su propósito es verificar que lo entregado cumple exactamente lo prometido.

**Un solo punto de desalineación es No-Go.**

---

## Quién valida

El validador es el **TL responsable de la comunicación con el cliente** — no el equipo que desarrolló. La separación es intencional: quien construyó no puede ser quien valida la entrega.

| Rol | Responsabilidad en el gateway |
|---|---|
| TL (comunicación con cliente) | Ejecuta el checklist, conduce la demo, firma el Go/No-Go |
| Desarrollador / equipo | Prepara el ambiente, los artefactos y la demo; disponible para preguntas |
| PM | Co-firma el Go junto al TL antes del merge a `master` |

---

## Cuándo ocurre

Después de que la rama `releases/v<version>` ha sido validada en el ambiente interno de QA de SCISA, antes del merge a `master`.

```
development → releases/v<x> → [QA interno SCISA] → Gateway de entrega → master
```

---

## Checklist de Go/No-Go

### Consistencia en tres puntas

La propuesta, la documentación en el repo y el software funcionando deben decir lo mismo. Cualquier desalineación entre los tres es No-Go.

- [ ] Los escenarios BDD de la propuesta están implementados y documentados en `/docs/features/`
- [ ] El diagrama de arquitectura en el repo refleja la solución entregada
- [ ] El manual de usuario describe exactamente lo que el usuario puede hacer en el sistema actual
- [ ] Si hubo requerimientos no funcionales o benchmarks en la propuesta: están ejecutados con evidencia adjunta

### Demo en vivo

- [ ] La demo se realiza sobre el ambiente publicado de QA (no sobre máquina local)
- [ ] Cubre todos los flujos principales de la propuesta
- [ ] Incluye al menos un caso de error relevante por flujo crítico

### Artefactos de entrega

- [ ] CHANGELOG actualizado con todas las versiones del release
- [ ] ADRs creados para decisiones arquitectónicas tomadas durante el proyecto
- [ ] Manual de usuario actualizado y consistente con el sistema entregado
- [ ] Diagrama de arquitectura actualizado en el repo

---

## Resultado

| Resultado | Condición | Siguiente paso |
|---|---|---|
| **Go** | Checklist 100% verde + demo exitosa | TL y PM firman; se procede al merge a `master` |
| **No-Go** | Cualquier punto rojo o desalineación detectada | Se documenta el punto de falla, se asigna responsable y fecha de resolución; el merge queda bloqueado |

Un No-Go no es un fracaso — es el sistema funcionando correctamente. Lo que no se puede hacer es ignorar el punto de falla y hacer el merge de todas formas.
