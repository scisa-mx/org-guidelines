# KPI de Entregas — Tasa de Fallos

Este documento define el KPI de calidad en entregas, el proceso de registro de incidencias y la cadencia de revisión. Su propósito es hacer visible si el proceso de entrega está mejorando o deteriorándose, y señalar con precisión en qué etapa del proceso se originan los fallos.

---

## KPI principal

```
Tasa de fallos = intentos fallidos / total de intentos de entrega
Meta: aproximarse a 0%
```

Un "intento de entrega" es cada ejecución del Gateway de entrega. Si un mismo release falla y se vuelve a intentar, son dos intentos — ocultarlo suavizaría el dato.

### Desagregación por etapa de origen

El KPI se desglosa según dónde se originó el fallo:

| Etapa | Qué indica | Qué proceso revisar |
|---|---|---|
| **Análisis** | Requerimiento mal definido, BDD ambiguos, ACs incompletos | Checklist de propuesta (`05.11`), DoR (`05.3`) |
| **Desarrollo** | Análisis correcto, implementación incorrecta o defecto introducido | DoD (`05.5`), code review |
| **Gateway** | Análisis y desarrollo correctos, el checklist no detectó la desalineación | Gateway de entrega (`05.12`) |

Si la mayoría de incidencias se concentran en una etapa, ese es el proceso que necesita mejora — no el equipo.

---

## Tipos de incidencia

### Incidencia de gateway
Ocurre cuando el Gateway de entrega produce un No-Go — el fallo se detecta internamente antes de que el cliente lo vea. Gravedad media: el sistema de control funcionó.

### Incidencia post-entrega
Ocurre cuando el cliente encuentra un problema después de que el release pasó el gateway. Gravedad alta: el gateway falló en detectar algo.

**Límite crítico:** solo es incidencia post-entrega si lo entregado contradice algo explícito en la propuesta aprobada (BDD scenarios, requerimientos no funcionales, manual de usuario). Si el cliente solicita algo que no estaba en la propuesta, es un cambio de alcance, no una incidencia.

---

## Registro de incidencias

Las incidencias se registran en **ClickUp** como tareas con los siguientes campos:

| Campo | Valores posibles |
|---|---|
| **Tipo** | Gateway / Post-entrega |
| **Proyecto / cliente** | Nombre del proyecto y cliente |
| **Release afectado** | `v<major>.<minor>.<patch>` |
| **Fecha del intento** | Fecha de la ejecución del gateway |
| **Etapa de origen** | Análisis / Desarrollo / Gateway |
| **Descripción del fallo** | Qué salió mal exactamente |
| **Causa raíz** | Por qué ocurrió; qué punto del proceso falló o no existía |
| **Resolución** | Qué se hizo para corregirlo |
| **¿Se actualizó el proceso?** | Sí / No |

### Cuándo se llena la causa raíz

- **Incidencia de gateway** — se llena en el momento del No-Go. El TL que ejecuta el gateway identifica el punto que falló y lo registra al abrir la tarea.
- **Incidencia post-entrega** — requiere análisis retrospectivo. El campo "Causa raíz" queda vacío hasta completar el análisis. El análisis debe realizarse dentro de los 5 días hábiles siguientes al reporte del cliente, con participación del TL y el responsable del desarrollo.

---

## Cadencia de revisión

### Por release
Al cerrar cada release se revisa la tasa de fallos del mismo: cuántos intentos se necesitaron y en qué etapa se originaron. Permite intervención inmediata si hay un patrón recurrente.

### Trimestral
Revisión acumulada cross-proyecto y cross-cliente. Es el momento donde el equipo decide si algún proceso se actualiza en respuesta a los datos. Si la tasa de fallos no bajó en el trimestre, se identifica la etapa dominante y se propone una mejora concreta al proceso correspondiente.

La revisión trimestral produce uno de tres resultados:
1. **Sin cambios** — la tasa está mejorando o estable en nivel aceptable
2. **Actualización de proceso** — se modifica el checklist de propuesta, DoD o Gateway de entrega según la etapa dominante
3. **Alerta** — la tasa subió; se convoca análisis extraordinario
