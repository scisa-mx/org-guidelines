# Guía de Revisiones Colaborativas
Hay que tener en cuenta que las revisiones colaborativas no son auditorías, no son exámenes, y no son una validación de jerarquía. Establecer un estándar claro para revisiones de código colaborativas, enfocadas en:

- Calidad técnica
- Transferencia de conocimiento
- Consistencia arquitectónica
- Mejora continua

## ¿Qué son las revisiones colaborativas?

Una revisión colaborativa es un proceso donde:

- El código se revisa entre pares
- El objetivo es mejorar el resultado final
- Se prioriza el aprendizaje compartido
- La responsabilidad del código es del equipo, no individual

## ¿Qué NO son las revisiones colaborativas?

No son:

- Un filtro burocrático
- Un juicio personal
- Un espacio para imponer estilos individuales
- Una búsqueda de culpables

No deben:

- Bloquear PRs por preferencias personales
- Convertirse en discusiones sin salida
- Reescribir código solo “porque sí”

## Principios base

### Colaboración sobre autoridad
La revisión no depende del seniority, el mejor argumento gana, no el rol.

### El código es el sujeto de revisión
Se revisa el código, no al autor, el feedback debe ser técnico y objetivo.

### Responsabilidad compartida
El código aprobado es responsabilidad del equipo, no existen “islas” de conocimiento. Si algo falla y paso por una revisión, el equipo lo asume. Es más eficiente asumir responsabildilidad compartida que buscar culpables. Se revisa, se planea y se abarca el problema desde el equipo el la busqueda de conocimiento.

### Feedback claro y accionable
Cada comentario debe:

- Tener contexto
- Tener razón técnica
- Proponer mejora

## ¿Qué se revisa en una revisión colaborativa?

- **Diseño y arquitectura**: SRP, SOLID, separación de capas, acoplamiento innecesario, etc.
- **Legibilidad y claridad**: Naming, tamaño de métodos, que los flujos sean entendibles.
- **Riesgos técnicos**: Null handling, seguridad, manejo de errores.
- **Consistencia**: Estándares de codificación y logging, patrones arquitectónicos, convenciones del proyecto.

## DOs/DON'Ts

### DOs

- Comentar con respeto
- Hacer preguntas abiertas
- Proponer alternativas
- Reconocer buenas decisiones

### DON'Ts

- Usar sarcasmo
- Bloquear sin razón técnica
- Imponer preferencias personales
- Aprobar sin entender
