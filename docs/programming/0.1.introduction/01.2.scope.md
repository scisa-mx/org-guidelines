# Alcance del Documento

Este documento aplica a todos los que contribuyen al desarrollo de software en SCISA, independientemente del rol o nivel de experiencia. No es exclusivo de los desarrolladores — cualquiera que participe en el ciclo de vida del software tiene responsabilidades aquí.

---

## ¿A quién aplica?

| Rol | Aplica |
|---|---|
| **Desarrolladores** (Junior, Mid, Senior) | Sí — es el núcleo de su práctica diaria |
| **Tech Leads / Arquitectos** | Sí — además de cumplirlos, son responsables de hacer que se cumplan |
| **Analistas de Negocio** | Parcialmente — secciones de documentación, DoR/DoD y comunicación |
| **QA / Testers** | Parcialmente — secciones de pruebas, criterios de aceptación y flujo de trabajo |
| **DevOps / Infraestructura** | Parcialmente — secciones de CI/CD, seguridad y scripts |

> Si participas en la creación, revisión o entrega de software en SCISA, algo de este documento te afecta directamente.

---

## ¿Qué cubre?

Este documento establece los lineamientos para:

- **Estándares de código**: estilo, convenciones, calidad y revisión.
- **Flujo de trabajo**: cómo se trabajan features, bugfixes y hotfixes; cómo se hace un PR.
- **Control de versiones**: ramas, commits, merging y roles en el repositorio.
- **Documentación**: qué documentar, dónde y cuándo.
- **Seguridad**: manejo de credenciales, validaciones y dependencias.
- **Comunicación**: cómo interactuar con el equipo, con analistas y con QA.
- **Herramientas**: IDE, linters, entornos locales y configuraciones estándar.
- **CI/CD**: pipelines, releases y pruebas automáticas.
- **Crecimiento**: rutas de aprendizaje y expectativas por nivel.

---

## ¿Qué NO cubre?

- Decisiones de arquitectura específicas por proyecto — esas van en los ADRs de cada repositorio.
- Configuraciones de infraestructura de producción — consulta con el equipo de DevOps.
- Reglas de negocio del dominio — esas viven en la documentación funcional de cada sistema.

---

## Aplicación por proyecto

Las reglas de este documento son el **mínimo base**. Cada equipo o proyecto puede establecer lineamientos adicionales más específicos, siempre que no contradigan lo definido aquí. Si existe un conflicto, este documento prevalece a menos que el Tech Lead del proyecto haya documentado explícitamente la excepción y su justificación.
