# Criterios para Definition of Ready

El **Definition of Ready** es lo que necesitas **antes** de empezar a trabajar para no perder tiempo en reuniones infinitas o suposiciones. Aquí tienes lo que yo agregaría para blindar tu proceso:

| Elemento | ¿Por qué es importante? |
| --- | --- |
| **Criterios de Aceptación (AC) claros** | Si no hay una lista de "Check/No Check", no hay forma de saber si terminaste. |
| **Diseños/Mocks Finales** | Evita el "esto no se ve como yo pensaba". Si hay UI, debe estar aprobada en Figma/Adobe XD. |
| **Dependencias Identificadas** | ¿Necesitas un endpoint de otro equipo? Si no está listo, el ticket no está "Ready". |
| **Contexto de Negocio** | Saber **para qué** sirve lo que programas ayuda a tomar mejores decisiones técnicas. |
| **Datos de Prueba disponibles** | ¿Tienes acceso a la base de datos o mocks de datos para probar escenarios reales? |
