# Guía de Estándares de Documentación

## Introducción

Este documento establece los lineamientos y estándares que los equipos de desarrollo deben seguir para mantener una documentación estructurada, consistente y actualizada. La documentación es parte integral del ciclo de desarrollo y no un entregable opcional; su ausencia bloquea la integración de código al repositorio.

La documentación en SCISA se divide en tres tipos con propósitos, audiencias y repositorios completamente independientes:

| Tipo | Audiencia | Ubicación |
|---|---|---|
| Documentación Técnica (Features BDD) | Equipo de desarrollo | Repositorio del proyecto (`/docs/features/`) |
| Architecture Decision Records (ADR) | Equipo de ingeniería | Repositorio del proyecto (`/docs/adr/`) |
| Manuales de Usuario | Cliente final | Repositorio Docusaurus independiente |

---

## Parte I — Documentación Técnica (Features BDD)

### 1.1 Propósito

La documentación técnica tiene como objetivo describir el comportamiento esperado de cada feature del sistema desde una perspectiva funcional y técnica, usando la metodología BDD (Behavior-Driven Development). Esta documentación vive junto al código y evoluciona con él.

### 1.2 Ubicación en el repositorio

Toda la documentación técnica reside dentro del repositorio del proyecto bajo la siguiente estructura de directorios:

```
/docs
└── features/
    ├── <usuarios>/
    │   └── <nombre-del-feature-especifico>.md
    │   └── <nombre-del-feature-especifico2>.md
    ├── <feature>/
    │   └── <nombre-del-feature-especifico>.md
    │   └── <nombre-del-feature-especifico2>.md
    │   └── <nombre-del-feature-especifico3>.md
    └── ...
```

Cada feature tiene su propia carpeta nombrada en `kebab-case`, descriptiva y consistente con el nombre usado en el backlog y los tickets.

### 1.3 Contenido de cada feature

Dentro de cada feature se deben incluir los siguientes puntos:

- **Descripción general:** Qué problema resuelve el feature y cuál es su alcance.
- **Actores involucrados:** Quiénes interactúan con el feature (roles de usuario, sistemas externos, etc.).
- **Precondiciones:** Estado del sistema necesario para que el feature opere correctamente.
- **Dependencias:** Otros features, servicios o módulos relacionados.
- **Notas técnicas relevantes:** Restricciones o consideraciones de implementación.

#### `<nombre-del-feature>.feature` — Escenarios BDD

Este archivo contiene los escenarios escritos en lenguaje Gherkin, siguiendo la estructura estándar `Given / When / Then`. Cada archivo `.feature` debe cubrir todos los escenarios relevantes del feature, incluyendo flujos felices y casos borde.

**Ejemplo de estructura:**

```gherkin
Feature: Autenticación de usuario

  Como usuario registrado
  Quiero poder iniciar sesión con mis credenciales
  Para acceder al sistema de forma segura

  Scenario: Inicio de sesión exitoso
    Given el usuario "carlos@scisa.com.mx" está registrado en el sistema
    When el usuario ingresa su contraseña correcta
    Then el sistema redirige al dashboard principal
    And se genera un token de sesión válido

  Scenario: Credenciales incorrectas
    Given el usuario "carlos@scisa.com.mx" está registrado en el sistema
    When el usuario ingresa una contraseña incorrecta
    Then el sistema muestra un mensaje de error
    And el intento de inicio de sesión queda registrado en los logs
```

### 1.4 Regla de protección de rama — Documentación obligatoria en PRs

Para garantizar que la documentación se mantenga actualizada con cada cambio en el código, el repositorio debe configurar una regla que bloquee el merge de un Pull Request cuando se detecte la adición o modificación de código relacionado con un feature sin la documentación correspondiente.

#### Mecanismo de validación

Se implementará un pipeline de Azure DevOps que se ejecute automáticamente sobre cada PR hacia las ramas protegidas. La lógica es directa: si el PR no toca ningún archivo dentro de `docs/features/`, el pipeline falla y el merge queda bloqueado.

Para que esto sea efectivo, el repositorio debe tener configurada una **Branch Policy** en Azure DevOps con las siguientes opciones activas:

- **Require a minimum number of reviewers** (al menos 1).
- **Check for linked work items** (opcional pero recomendado).
- **Build validation** apuntando al pipeline de validación de documentación — con la opción **"Block if the build fails"** activada.

#### Ejemplo de workflow

```yaml
trigger: none

pr:
  branches:
    include:
      - main
      - development

pool:
  vmImage: ubuntu-latest

steps:
  - checkout: self
    fetchDepth: 0

  - script: |
      echo "Verificando cambios en docs/features/..."

      CHANGED=$(git diff --name-only origin/$(System.PullRequest.TargetBranch)...HEAD)
      echo "Archivos modificados en este PR:"
      echo "$CHANGED"

      HAS_DOCS=$(echo "$CHANGED" | grep -c "^docs/features/" || true)

      if [ "$HAS_DOCS" -eq 0 ]; then
        echo ""
        echo "##[error] Este PR no incluye cambios en docs/features/."
        echo "##[error] Todo PR debe documentar el feature correspondiente."
        echo "##[error] Agrega o actualiza los archivos en docs/features/<nombre-del-feature>/ antes de solicitar el merge."
        exit 1
      fi

      echo "Documentación presente. Validación exitosa."
    displayName: "Validar presencia de documentación en docs/features"
```

> **Nota:** Esta configuración es de ejemplo y no refleja ningún repositorio o proyecto real.

### 1.5 Responsabilidades

| Rol | Responsabilidad |
|---|---|
| Desarrollador | Crear o actualizar la documentación del feature antes de abrir el PR |
| Reviewer del PR | Verificar que el contenido de la documentación sea coherente con la implementación |
| Tech Lead | Mantener el estándar de calidad de la documentación técnica |

## Parte II — Manuales de Usuario

### 2.1 Propósito

Los manuales de usuario tienen como objetivo guiar al cliente final en el uso de los productos de SCISA. Esta documentación debe ser clara, visual y estructurada por producto y módulo, de forma que cualquier usuario pueda encontrar rápidamente la información que necesita sin conocimiento técnico previo.

### 2.2 Tecnología y repositorio

Los manuales de usuario se alojan en un repositorio independiente basado en **Docusaurus**, un generador de sitios de documentación estática. Cada cliente tiene su propio repositorio de documentación, completamente separado del repositorio de código.

### 2.3 Estructura del repositorio Docusaurus

```
/
├── docs/
│   └── <producto>/
│       └── <modulo>/
│           ├── intro.md
│           ├── <manual-de-usuario>.md
│           └── ...
├── static/
│   └── img/
├── docusaurus.config.js
├── sidebars.js
├── Dockerfile
└── .github/
    └── workflows/
        └── deploy.yml
```

La jerarquía de contenido sigue tres niveles:

1. **Producto** — Corresponde a un producto de SCISA (por ejemplo, `pld`, `siglo`, etc).
2. **Módulo** — Agrupación funcional dentro del producto (por ejemplo, `configuracion`, `reportes`, `usuarios`).
3. **Manual** — Documento específico con la guía de uso de una funcionalidad dentro del módulo.

### 2.4 Ciclo de actualización

El repositorio de documentación de usuario se actualiza en sincronía con cada entregable de desarrollo. El proceso es el siguiente:

1. Al cerrar un entregable o sprint, el equipo identifica qué funcionalidades visibles para el usuario fueron modificadas o añadidas.
2. Se abre un PR en el repositorio de Docusaurus con los cambios correspondientes en el manual de usuario.
3. El PR es revisado y mergeado a la rama principal.
4. El pipeline de CI/CD publica automáticamente los cambios en el sitio de documentación.

Este proceso garantiza que la documentación de usuario siempre refleje el estado actual del producto entregado.

### 2.5 Publicación automatizada en Google Cloud con Docker

El sitio de Docusaurus se publica como un contenedor Docker en **Google Cloud Run**, lo que permite una publicación sin servidor, escalable y automatizada.

### 2.6 Responsabilidades

| Rol | Responsabilidad |
|---|---|
| Desarrollador / Analista | Redactar los cambios en el manual al cerrar un entregable |
| Responsable de producto | Revisar que el manual refleje correctamente la funcionalidad entregada |
| DevOps / Infraestructura | Mantener el pipeline y la infraestructura de publicación en GCP |

## Parte III — Architecture Decision Records (ADR)

### 3.1 ¿Qué es un ADR?

Un Architecture Decision Record es un documento corto que registra una decisión arquitectónica significativa tomada durante el desarrollo de un sistema: qué se decidió, por qué se decidió así, qué alternativas se consideraron y cuáles son las consecuencias de esa decisión. No documenta cómo funciona el sistema, sino el razonamiento detrás de las elecciones de diseño que le dieron forma.

Los ADRs son especialmente valiosos porque las decisiones de arquitectura suelen perderse con el tiempo. A medida que el equipo cambia o el sistema crece, nadie recuerda por qué se eligió una tecnología, por qué se descartó un patrón, o qué problema se intentaba resolver. Un ADR deja ese contexto escrito y accesible para cualquier persona que trabaje en el proyecto, ahora o en el futuro.

Cada ADR es inmutable una vez aceptado: no se edita, se supercede. Si una decisión cambia, se crea un nuevo ADR que referencia y reemplaza al anterior, preservando el historial completo del razonamiento del equipo.

### 3.2 Ubicación en el repositorio

Todos los ADRs residen dentro del repositorio del proyecto bajo la siguiente ruta:

```
/docs
└── adr/
    ├── 0001-<titulo-de-la-decision>.md
    ├── 0002-<titulo-de-la-decision>.md
    └── ...
```

Los archivos se nombran con un número secuencial de cuatro dígitos seguido de un título descriptivo en `kebab-case`. El número es permanente e irrepetible; garantiza que el orden cronológico de las decisiones sea siempre legible desde el sistema de archivos.

## Resumen comparativo

| Criterio | Documentación Técnica | Manuales de Usuario | ADR |
|---|---|---|---|
| Audiencia | Equipo de desarrollo | Cliente final | Equipo de ingeniería |
| Repositorio | Dentro del repo del proyecto | Repositorio Docusaurus independiente | Dentro del repo del proyecto |
| Ubicación | `/docs/features/<feature>/` | `/docs/<producto>/<modulo>/` | `/docs/adr/` |

| Validación | Check obligatorio en PR | Revisión en PR del repositorio de docs | Revisión por Tech Lead |

## Puntos Pendientes

Los siguientes puntos han sido identificados como decisiones no resueltas que deben definirse antes de que los equipos adopten este estándar de forma completa.

### P1 — Template específica para documentación técnica

Aún no se ha definido una plantilla oficial para los archivos `README.md` y `.feature` dentro de `/docs/features/`. Es necesario diseñar y acordar una template concreta que estandarice:

- Las secciones obligatorias y opcionales del `README.md` por feature.
- La convención de nombrado de Scenarios dentro del archivo `.feature`.
- El nivel de detalle esperado en los pasos `Given / When / Then`.
- Criterios de aceptación mínimos para que una documentación técnica sea considerada válida en el PR.

**Acción requerida:** Definir y publicar las templates en un repositorio de referencia o en la wiki del equipo antes del inicio del siguiente ciclo de desarrollo.

### P2 — Modelo de acceso para los Manuales de Usuario

No se ha tomado una decisión respecto al nivel de acceso del sitio de Docusaurus. Existen tres escenarios posibles:

- **Público:** El sitio es accesible sin autenticación para cualquier persona con la URL.
- **Privado con autenticación:** El sitio requiere login (por ejemplo, mediante un Identity Provider como Keycloak, Google OAuth o similar) para ser consultado.
- **Dependiente del cliente:** Algunos clientes tienen el sitio público y otros privado, lo que implicaría una instancia de Cloud Run por cliente o un mecanismo de control de acceso por tenant.

Esta decisión impacta directamente en la arquitectura del pipeline de publicación, el Dockerfile, la configuración de Cloud Run y los costos de infraestructura.

**Acción requerida:** Definir el modelo de acceso con los responsables de producto y el equipo de infraestructura, y actualizar este documento y el pipeline de despliegue en consecuencia.

### P3 — Barra de búsqueda en template de Docusaurus

Falta definir e integrar la barra de búsqueda dentro de la template base de Docusaurus. Docusaurus soporta búsqueda local mediante el plugin `@docusaurus/plugin-search-local` o búsqueda avanzada vía Algolia DocSearch.

**Acción requerida:** Evaluar qué opción de búsqueda se ajusta mejor al volumen de documentación y al modelo de acceso elegido (P2), e integrarlo en la template base antes del primer despliegue.
