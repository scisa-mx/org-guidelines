Escribir documentación no es llenar folios de texto que nadie leerá; es un acto de empatía con tu "yo" del futuro y con tus compañeros. La meta es que el conocimiento sea **descubrible, útil y, sobre todo, vivo.**

Aquí tienes la guía estructurada para mantener esa "documentación activa" que mencionas:

---

## 1. El Código como Primera Fuente de Verdad

Antes de escribir un archivo `.md`, asegúrate de que tu código no necesite traducción.

* **Nombres semánticos:** `calculate_final_price()` es mejor que `calc()`.
* **El "Por qué" vs. el "Qué":** El código dice *qué* estás haciendo. Los comentarios en el código solo deben usarse para explicar el *porqué* (decisiones técnicas extrañas, parches por bugs externos o lógica de negocio compleja).
* **Self-Documenting:** Si una función es demasiado compleja para explicarla en una línea, divídela.

---

## 2. ¿Dónde guardar el conocimiento?

No toda la información pertenece al mismo lugar. Usa este criterio para decidir:

| Tipo de Información | Destino | Ejemplo |
| --- | --- | --- |
| **Aprendizaje General** | [Stack Overflow Teams](https://stackoverflowteams.com/c/scisa) | "Cómo configurar Docker con M1", "Solución a error X de React". |
| **Lógica del Negocio** | Repositorio (`/docs` o `README.md`) | "Criterios para calcular el IVA en este cliente", "Flujo de checkout". |
| **Arquitectura / Visuales** | Repositorio (Archivos `.mermaid` o imágenes) | Diagramas de secuencia, entidad-relación, flujos de usuario. |
| **Configuración Local** | `README.md` | Variables de entorno, pasos para el `npm install`. |

---

## 3. Documentación Visual (Diagramas y Flujos)

Un diagrama bien hecho ahorra 10 reuniones de 30 minutos.

* **Diagramas como Código:** Siempre que puedas, usa herramientas como **Mermaid.js** dentro de tus archivos Markdown. Esto permite que el diagrama viva en el repo y se pueda editar sin herramientas externas.
* **Capturas de pantalla:** Si la interfaz es compleja, una captura con flechas vale oro. Guárdalas en una carpeta `/assets` dentro de tu carpeta de documentación.

---

## 4. El Concepto de "Documentación Activa"

Para que la documentación no muera y se vuelva obsoleta ("rot"), sigue estos principios:

* **Docs-as-Code:** Trata la documentación como tratas el código. Si cambias una funcionalidad, el PR (Pull Request) **no se aprueba** si no incluye la actualización de los docs relacionados.
* **Elimina la fricción:** Si algo te costó más de 15 minutos entenderlo o configurarlo, **debe ser documentado inmediatamente.** Es el momento en que el dolor está fresco y serás más preciso.
* **README Minimalista:** Todo repositorio debe tener un `README.md` que responda:
1. ¿Qué hace este proyecto?
2. ¿Cómo lo hago funcionar en mi máquina? (Setup rápido).
3. ¿Dónde están los flujos principales? (Links a otros documentos).



---

## 5. Checklist de Documentación para tu PR

Antes de darle a "Submit", pregúntate:

1. [ ] ¿Actualicé el `README` si agregué nuevas variables de entorno (`.env`)?
2. [ ] ¿Si alguien nuevo entra hoy al equipo, puede entender este flujo con lo que escribí?
3. [ ] ¿Subí los diagramas que dibujé en la pizarra o en Figma al repo?
4. [ ] ¿Lo que aprendí de este bug ya está en Stack Overflow Teams para que otros no sufran lo mismo?

> **Nota:** La documentación activa es la que evoluciona. Si encuentras un doc desactualizado, bórralo o corrígelo en el momento. No lo dejes ahí "para luego".
