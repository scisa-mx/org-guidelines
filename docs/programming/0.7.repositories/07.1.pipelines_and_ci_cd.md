# Pipeline de CI/CD

## ¿Qué es CI/CD?
CI/CD es un conjunto de prácticas de ingeniería de software que buscan que el código:

Se integre correctamente
Se valide automáticamente
Esté siempre en un estado entregable

CI/CD no es un pipeline, no es Azure DevOps, no es GitHub Actions.

CI/CD se compone de dos conceptos
CI – Continuous Integration
CD – Continuous Delivery / Continuous Deployment

Cada uno resuelve un problema distinto: 

### Continuous Integration (CI)
Continuous Integration es la práctica de integrar cambios de código de forma frecuente, validándolos automáticamente. Cada vez que alguien cambia código:

- Se integra con el código existente
- Se valida que no rompa nada
- Se detectan errores lo antes posible

Todo esto resuelve una serie de problemas muy especificos:
- Se dejan de tener cambios muy grandes que puedan romper cosas y nos saber exactamente que rompio que.
- Integraciones tardías.
- Bugs detectados al final del desarrollo.

### Continuous Deployment (CD)
Continuous Deployment significa que el software siempre está en un estado listo para ser liberado, y pueda ser liberado automaticamente.
Es decir:

- Cada cambio pasa por validaciones
- El artefacto generado es desplegable
- La decisión de liberar es humana
- Todo cambio que pasa CI
- Se despliega automáticamente
- Sin intervención humana

Nuevamente esto resuelve una serie de problemas, como:

- Releases manuales
- Pasos frágiles
- Deploys con miedo o hechos tardiamente
- Dependencia de personas clave

## ¿Cómo se entrelaza CI/CD con los pipelines?
Piensa en CI/CD como la filosofía y en los pipelines como la herramienta concreta que la hace realidad. Esta metodología responde un "qué" y al "por qué", y el pipeline es una herramienta estandar que nos ayuda con el "cómo" y "cuándo".

### Relación conceptual (vista global)

#### CI (Continuous Integration)

Busca que:

Los cambios se integren frecuentemente
El código se compile, pruebe y valide automáticamente
Se detecten errores lo antes posible

Aquí es donde aparece el pipeline de CI.

Un pipeline de CI típicamente:
- Se dispara por un push o pull request
- Restaura dependencias
- Compila
- Ejecuta pruebas
- Publica artefactos (opcional)
- Sin pipeline, CI sería solo una buena intención.

#### CD (Continuous Delivery / Deployment)

Busca que:

El software esté siempre listo para desplegar
El despliegue sea repetible, confiable y automatizado
Se reduzcan errores humanos

**Aquí entran los pipelines de CD.**

Un pipeline de CD:

- Toma artefactos/contenedores generados por CI
- Despliega a ambientes (dev, qa, prod)
- Aplica configuraciones por entorno
- Puede requerir aprobaciones

Pipeline = la materialización del CI/CD

Un pipeline es básicamente:

Una definición automatizada de pasos que se ejecutan bajo ciertas condiciones.

Incluye:

- Triggers (cuándo se ejecuta)
- Stages (CI, CD, etc.)
- Jobs (bloques de trabajo)
- Steps (comandos concretos)

Sin CI/CD:

Un pipeline es solo automatización suelta
Sin pipelines:

***CI/CD no se puede ejecutar de forma consistente***

## Enfoque específico: Azure Pipelines

Azure Pipelines es la implementación de Microsoft para llevar CI/CD a la práctica.

¿Qué aporta Azure al concepto?

Azure Pipelines:

- Permite definir CI y CD en un solo flujo
- Usa pipelines declarativos (YAML)

Se integra directo con:

- Repos Git (Azure Repos, GitHub, Bitbucket)
- Docker
- Kubernetes
- Azure App Services, VMs, etc.

Actualmente a la hora de escribir este documento es el estándar en la organización y su utilización debe de entenderse como un paso natural y necesario en el desarrollo de cualquier repositorio.
