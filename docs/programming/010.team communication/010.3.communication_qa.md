# Guía – Etapa de QA como Responsabilidad del Desarrollador

Definir una etapa formal de QA dentro del flujo de desarrollo cuando no existe un equipo de QA dedicado.

El propósito es asegurar calidad, estabilidad y cumplimiento funcional antes de considerar un ticket como terminado.

En este modelo:

El desarrollador no solo construye.  
También valida.


## Principio Base

Si no hay equipo de QA, la calidad no desaparece.  
Se traslada como responsabilidad explícita al desarrollador y al equipo.

La etapa de QA no es opcional.  
Es una fase obligatoria antes de cerrar o mover un ticket a revisión.

## ¿Qué significa tener una “Etapa de QA”?

Significa que, después de terminar la implementación:

1. El desarrollador cambia mentalidad de “constructor” a “validador”.
2. Prueba el sistema como si fuera usuario final.
3. Ejecuta escenarios positivos y negativos.
4. Verifica reglas de negocio completas.
5. Documenta evidencias si es necesario.

No es solo “probar que funciona”.  
Es intentar romper lo que se construyó.



## Flujo Recomendado

Desarrollo → Auto QA → Ajustes → PR → Revisión → Merge → Deploy → Etapa de QA

La etapa de Auto QA ocurre antes del PR.



## 1. Validación Funcional

Antes de marcar como listo:

- Confirmar que se cumplen todos los criterios de aceptación.
- Probar flujo principal completo.
- Validar escenarios alternos.
- Revisar mensajes de error.
- Confirmar que no se rompió funcionalidad existente.

Preguntas clave:

- ¿Qué pasa si el usuario no llena un campo obligatorio?
- ¿Qué pasa si el servicio externo falla?
- ¿Qué pasa si no tiene permisos?
- ¿Qué pasa con datos límite o inválidos?



## 2. Validación Técnica

Revisar:

- Logs innecesarios.
- Manejo correcto de errores.
- Performance básico.
- Validaciones de seguridad.
- Manejo adecuado de excepciones.
- Impacto en otros módulos.



## 3. Validación de Regla de Negocio

No asumir que lo técnico es suficiente.

Confirmar:

- ¿La lógica cumple exactamente con lo solicitado?
- ¿Se respetan reglas especiales?
- ¿Hay condiciones implícitas que deben validarse?
- ¿El comportamiento coincide con lo acordado con negocio?



## 4. Checklist Mínimo de Auto QA

Antes de abrir PR:

- Probé flujo principal.
- Probé escenarios inválidos.
- Probé permisos/roles.
- Validé mensajes de error.
- Revisé logs.
- Confirmé que no hay vulnerabilidades nuevas.
- Revisé dependencias afectadas.

Si alguno no se cumple, no está listo.


## 5. Evidencia Recomendada

Cuando el cambio es relevante:

- Adjuntar capturas.
- Documentar pasos de prueba.
- Describir casos validados.
- Agregar notas en el PR.

Esto facilita revisión por parte del equipo.
