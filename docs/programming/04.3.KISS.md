# KISS - Keep It Simple, Stupid

En el mundo del desarrollo de software, es muy común caer en la tentación de crear soluciones complejas, llenas de abstracciones, patrones y capas “por si algún día se necesitan”.

Aqui es donde entra la importancia de las buenas practicas de programación, y una de las más importantes es el principio KISS.

**¿Qué significa KISS?**
KISS es el acrónimo de: `Keep It Simple, Stupid`.

---

## ¿Cuál es la idea central de KISS?

En diseño de software, esto se traduce en escribir código que:
- Sea fácil de leer y entender.
- Minimiza el número de partes móviles (funciones, clases, métodos).
- Evita soluciones demasiado inteligentes que podrían confundir a futuros desarrolladores.

La complejidad innecesaria genera:
- Más bugs
- Más tiempo de mantenimiento
- Mayor curva de aprendizaje para nuevos desarrolladores

En pocas palabras: `La complejidad es un costo, no un logro`.

El objetivo no es sacrificar la funcionalidad, sino diseñar la solución más sencilla posible sin comprometer el rendimiento ni la usabilidad. Los sistemas sencillos son más fáciles de mantener, depurar y ampliar, lo cual es esencial en entornos de desarrollo dinámicos.

## ¿Por qué es importante el principio KISS?

- Mantenibilidad: El código simple es más fácil de mantener. Cuando el código es claro, los futuros desarrolladores (incluido tú mismo) pueden comprender la lógica sin tener que lidiar con patrones complejos. Una función clara de diez líneas es más fácil de solucionar y actualizar que una función de una sola línea, compacta pero densa.

- Legibilidad: El código simple es más legible. Esto es crucial en entornos colaborativos donde varios desarrolladores contribuyen a una base de código. El código legible permite que los equipos colaboren eficazmente y garantiza una entrega fluida.

- Depuración: Los errores son más fáciles de detectar en sistemas simples. Cuando el flujo de control es directo, identificar la causa de un error requiere menos tiempo. Los sistemas complejos suelen ocultar errores en partes oscuras del código, lo que convierte la depuración en una pesadilla.

- Escalabilidad: Los sistemas simples escalan con mayor eficiencia. Al no verse abrumado por una complejidad excesiva, es más fácil desarrollar sobre el código base existente. Unas bases sencillas permiten extensiones y mejoras flexibles.


## Ejemplos

### Ejemplo 1: Función simple vs función compleja

```csharp
public bool IsEven(int number)
{
    return number % 2 == 0 ? true : false;
}
```
Este código funciona, pero añade ruido visual y lógica redundante.

Una versión más simple sería:

```csharp
public bool IsEven(int number)
{
    return number % 2 == 0;
}
```

### Ejemplo 2: Evitar absrtacciones innecesarias

```csharp
public class NumberValidator
{
    public bool Validate(int number)
    {
        return new EvenNumberSpecification().IsSatisfiedBy(number);
    }
}
```
Este enfoque puede ser válido en escenarios complejos, pero para una validación simple es excesivo.

Una versión más simple sería:
```csharp
public bool IsEven(int number)
{
    return number % 2 == 0;
}
```

## KISS NO significa.

- Código rápido y desordenado
- Ignorar arquitectura
- No usar patrones
- “Luego lo arreglo”

Una frase muy común mientras desarrollas puede ser "¿Esto realmente necesita ser tan complejo?" y en la mayoria de los casos, la respuesta es no.
