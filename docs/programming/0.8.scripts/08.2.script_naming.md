# Estandarización del naming de scripts

La estandarización del nombre de los scripts puede parecer un detalle menor, pero en codebases medianos o grandes se convierte rápidamente en un factor crítico de mantenibilidad, seguridad y eficiencia operativa.

Un buen naming permite entender qué hace un script sin abrirlo, reduce errores humanos y facilita su uso dentro de pipelines, revisiones de código y procesos de automatización.

Desde una perspectiva DevSecOps, el nombre de un script también comunica riesgo: no es lo mismo ejecutar `deploy-prod.sh`que `run.sh`.

## El problema de no tener un estándar

Cuando no existe una convención clara, aparecen patrones peligrosos:

- Scripts llamados `test.sh`, `script2.sh`, `final.ps1`
- Nombres ambiguos como `run.sh` o `setup.ps1`
- Scripts con efectos destructivos mal identificados
- Dificultad para distinguir scripts locales, de CI o de producción
- Errores por ejecución accidental

Estos problemas no son teóricos: muchos incidentes ocurren porque alguien ejecutó el script equivocado.

## El nombre como contrato

Un script no solo ejecuta acciones, también expone una intención.
El nombre debe responder, de forma clara, a estas preguntas:

- ¿Qué hace?
- ¿Sobre qué actúa?
- ¿En qué contexto se ejecuta?
- ¿Tiene impacto destructivo?

Si el nombre no responde a eso, el script está mal nombrado.

## Convención general recomendada

La convención debe ser:

- Descriptiva
- Predecible
- Consistente
- Compatible con CI/CD

```text
<accion>-<objetivo>-<contexto>.<extensión>
```

Ejemplos:

- run-tests-ci.sh
- migrate-db-prod.ps1
- setup-env-local.ps1
- deploy-api-qa.sh
- cleanup-temp-files.sh

## Buenas prácticas

- Usar nombres largos pero claros
- Mantener una convención única en todo el repo
- Revisar naming en PR
- Diferenciar entorno y contexto
- Hacer explícito el impacto del script
- Alinear naming con pipelines y carpetas
